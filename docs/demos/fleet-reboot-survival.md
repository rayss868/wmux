# 함대 리부트 생존 — 수동 데모 시나리오 (J3 §5(b))

> 촬영·시연용 절차. 자동 판정식은 `scripts/j3-fleet-reboot-survival-demo.mjs`(데몬 상태 복원 실증)이고,
> 이 문서는 그 위 계층 — **워크스페이스·페인·diff 재열람**까지 사람 눈으로 확인하는 절차다.
> 문구 규율(전략 P7): 이 데모의 주장 범위는 "reboot-resilient agent sessions"까지다.
> "죽지 않는", "절대" 계열 표현은 데모·랜딩 어디에도 쓰지 않는다.

## 사전 준비

1. 데모용 git repo 1개(변경 여지가 있는 코드, 커밋 이력 존재).
2. wmux 데몬 모드 실행 확인(`wmux doctor` 또는 상태바 데몬 표시).
3. 에이전트 CLI(claude) 로그인 상태.

## 절차 (촬영 컷 단위)

| # | 컷 | 액션 | 관객이 보는 것 |
|---|----|------|---------------|
| 1 | fan-out | AgentToolbar → fan-out → 프롬프트 1개 입력, N=4, 태스크 title 4개 확인 → 실행 | 워크스페이스 4개가 사이드바에 생기고, 미션 채널 4개가 Channels 독에 등장 |
| 2 | 작업 확인 | 태스크 워크스페이스 2~3개를 순회 — 에이전트가 각자 worktree에서 작업 중 | 각 워크스페이스 페인의 cwd가 서로 다른 worktree 경로 |
| 3 | **강제 종료** | wmux 앱을 완전히 종료(⌘Q — 필요 시 데몬 프로세스도 kill로 강화 버전) | 화면이 닫힌다 |
| 4 | 재기동 | wmux 재실행 | **워크스페이스 4개가 그 자리에 복원** — 이름·페인 구성·cwd 유지 |
| 5 | 태스크 확인 | 태스크 목록(task.mission.list 소비 UI 또는 채널 독) 확인 | 4태스크 전원 open·물질화 필드 잔존, 미션 채널 4개 active·히스토리 유지 |
| 6 | 산출물 확인 | 태스크 하나의 diff 서피스 열기 | 재시작 전 에이전트가 만든 미커밋 변경이 diff에 그대로 |
| 7 | (마무리) | hunk 채택 1건 시연 → 타겟에 반영 | 여정 완결: fan-out → 리부트 생존 → 수확 |

## 판정 기준

- 컷 4~6에서 **수동 복구 조작 0회**(사람이 아무것도 다시 만들지 않음)이면 PASS.
- 에이전트 프로세스 자체는 재기동 후 재시작 대상이다(세션·상태가 남는 것이지 프로세스가 죽지 않는 것이 아님) — 시연 멘트에서 이 경계를 정직하게 말한다.

## 강화 버전(선택)

- 컷 3에서 ⌘Q 대신 `kill -9`(데몬 포함) 또는 OS 재부팅으로 대체 — 동일 판정 기준.
- 이 경우 세션 저장의 크래시-손실 창(≤5s) 안에 든 마지막 변경은 유실될 수 있음을 사전 고지(정직성 규율).
