## Problem Statement
Channels are agent-and-human multi-party rooms, but the UI has no way to **see who is in a channel**, **join** a channel an agent created, or **leave** one. The composer's `postMessageDaemon` already requires the caller to be a member (daemon `NOT_A_MEMBER`), and `events.poll` only fans a `channel.message` to workspaces in `recipientWorkspaceIds` (= members). So "can I (the human) participate" bottoms out at membership — and there is no membership UI. The drag-to-join prototype was pulled (UX panel: mouse-only WCAG 2.1.1 Level A failure, add-only trap, zero discoverability). This designs the discoverable, accessible, reversible membership loop the panel called the keystone.

## What Makes This Cool
After this, a human can open a channel an agent is running, click **+ member**, add their workspace, and start receiving the agents' live messages — the first time the "team chat for your agents" pitch is actually true end-to-end. And it makes a clean 2-party live-delivery test possible (currently impossible to even set up).

## Key finding that shapes the design (from the 2026-06-23 investigate + office-hours)
- Channel.message **delivery code is correct end-to-end** (traced every hop; `EventBus.poll` advances the cursor past the metadata flood, no resync loss). The blocker to participation is **membership**, not delivery.
- The member-identity model is **not broken**: schema is `(workspaceId, memberId)`; agents posting via MCP supply their own `member_id` ("lead"/"backend") = real per-agent membership. The renderer's fixed `memberId:'local-ui'` is the **human/UI** path, which is correct (the human is one entity per workspace).

## Decision (locked)
**Workspace = member for the UI path** (keep the existing UI memberId `'local-ui'` AS-IS). Agents stay per-agent via MCP. **No identity re-architecture.** Build roster + join + leave + keyboard path on top.
- **v1 join = PUBLIC channels only.** The "+ member" picker offers only public channels. Private-channel join = invite flow, deferred. (See Security note.)
- **Do NOT rename `'local-ui'`.** (Review caught: it's a persisted/wire value stored in daemon channel state as `memberId`/`createdBy`; renaming breaks `leave` for channels joined before the rename — daemon dedup/leave key on `memberId`, so `'human'` ≠ stored `'local-ui'` → `NOT_A_MEMBER` — and it touches Composer ×2, ChannelsPanel ×3, tests, and the dogfood, not "3 lines." Not zero-behavior. Out of scope.)

## Security note (pre-existing daemon gap — flag, separate fix)
`ChannelService.join()` (`src/daemon/channels/ChannelService.ts:512-554`) has **NO visibility gate** — it checks only `CHANNEL_NOT_FOUND` and `DUPLICATE_MEMBER`, never `channel.visibility`/`isVisibleTo`. So **any caller that knows a private `channelId` can join it** and unlock its history + live feed (reads ARE membership-scoped, so join is the escalation). This is independent of this UI. **v1 mitigations:** the membership UI offers public channels only (reduces the GUI surface), but the hole stays open to any pipe/MCP caller. **Separate security follow-up:** add an `isVisibleTo`/invite gate to daemon `join()`. Surfacing now (REPO_MODE solo, maintainer owns it).

## Approaches Considered

### Approach A: Roster popover from a header member-count button (minimal viable)
  Summary: ChannelView header gets a "👥 N" member-count button → click opens a popover listing members (name + workspace + human/agent), each removable; a "+ member" row opens a workspace picker.
  Effort: M   Risk: Low
  Pros: member count is always visible (discoverability) · one surface holds see+join+leave · smallest diff · keyboard/a11y native (buttons) · matches panel's "roster in ChannelView header" keystone.
  Cons: popover real-estate for long member lists (scroll).
  Reuses: ChannelViewContent header, channelMembers store, joinChannelOptimistic/leaveChannelOptimistic, channels:mutate-local.

### Approach B: Always-visible collapsible "Members" section in ChannelView
  Summary: a persistent Members strip (count + expandable list) above the composer, + a modal workspace picker for add.
  Effort: L   Risk: Med
  Pros: most discoverable (always shown).
  Cons: eats vertical space in an already-narrow dock; modal picker is heavier; more diff.

### Approach C: Inline member chips in the header (lateral, Slack-like)
  Summary: avatar/initials chips in the header; "+" chip adds, hover-✕ removes.
  Effort: M   Risk: Med
  Pros: compact, familiar.
  Cons: hover-✕ is not keyboard-friendly (repeats the a11y mistake); chips overflow in a 320px dock.

**RECOMMENDATION: Approach A.** Always-visible count = discoverability; popover holds the full see/join/leave loop; buttons = keyboard + screen-reader operable (fixes the WCAG 2.1.1 failure); smallest complete diff.

## Recommended Approach — build scope

**Store (channelsSlice):** (NB: neither `*Daemon` join/leave exists in the live tree — reconstruct, don't "revert")
- `joinChannelDaemon(channelId, member, workspaceId)` (mutateLocal `a2a.channel.join` → `joinChannelOptimistic`). `member = {workspaceId: self, memberId: 'local-ui', memberName}`.
- `leaveChannelDaemon(channelId, memberId, workspaceId)` — **self-only**, so no `member` object. mutateLocal `a2a.channel.leave` with `{channelId, workspaceId: self, memberId, verifiedWorkspaceId: self}` (daemon requires all four; `leave()` matches `m.workspaceId === verifiedWorkspaceId === self` — `ChannelService.ts:573`) → on ok `leaveChannelOptimistic(channelId, memberId)` (2-arg, matches `channelsSlice.ts:329`).
- Unit tests for both (success / NOT_A_MEMBER / DUPLICATE_MEMBER / bridge-missing).

**UI:**
- `ChannelMembersButton` + `ChannelMembersPopover` (new, pure views): count button in the ChannelViewContent header; popover lists `channelMembers[channelId]` (memberName, workspace label = `workspace.name`, human/agent tag). **Remove (✕) shows ONLY on the exact `(self workspaceId, self memberId)` row** — self-leave, not eject; "own workspace" is not enough (an agent in your workspace has a different memberId and daemon `leave` would `NOT_A_MEMBER`). A "+ member" row opens a workspace picker of **public-joinable** workspaces not already members → click to join.
- **On successful self-leave, if the left channel is `activeChannelId` → `setActiveChannel(null)`** (in scope at `ChannelView.tsx:254/257`). Else the dock shows a dead/blank pane (private channel reads go empty for a non-member). Review-caught; mandatory.
- **Archived channel: hide/disable "+ member"** (daemon `join`/`leave` don't check archived; post does — keep the surface consistent with read-only).
- **No resolvable self-workspace** (`!company && !activeWorkspaceId`): disable "+ member" and remove (daemon would `NOT_AUTHORIZED` on empty `verifiedWorkspaceId`).
- Wire into ChannelViewContent header (next to #name / close).
- Container resolves identity (`company?.ceoWorkspaceId ?? activeWorkspaceId`) for join/leave, mirrors ChannelsPanel. Workspace picker source = `useStore(s => s.workspaces)` (`workspaceSlice.ts:36`, has `id`+`name`).

**a11y:** count button, member rows, remove, +member, picker entries are all `<button>` with focus ring + aria-labels. Toasts already `role=status`/`aria-live`. No drag required → WCAG 2.1.1 satisfied.

**i18n:** `channels.members` ("Members"), `channels.addMember`, `channels.removeMember`, `channels.memberCount`, join/leave toasts (some exist from the reverted drag work — re-add).

## Resolved by spec review (2026-06-23, adversarial, scored 5→8 after fixes)
- Remove scope = **self-only** (matches daemon `leave()` capability; eject = new authz, deferred). Locked.
- Private join = **deferred** (UI public-only); daemon visibility gate = separate security follow-up (Security note).
- `'local-ui'` rename = **dropped** (persisted wire value; breaks pre-rename leave).
- Self-leave of active channel → **`setActiveChannel(null)`** (mandatory).
- `leaveChannelDaemon` = self-only `(channelId, memberId, workspaceId)`, no `member` object.
- Remove (✕) row = exact `(self ws, self memberId)` only.
- Workspace picker source = `useStore(s => s.workspaces)` (confirmed present).
